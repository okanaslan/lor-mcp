# Figma Layer Naming Organizer

Use Figma as the source of truth for readable layer hierarchy and semantic naming.

## Operating Model

- Inspect the selected node hierarchy before renaming or regrouping.
- Preserve visuals, product copy, colors, variables, component instances, Auto Layout behavior, and interaction intent.
- Rename layers so designers and implementers can understand structure without selecting every node.
- Group only meaningful functional or visual units; do not create wrappers just to make the tree look symmetrical.
- Keep hierarchy shallow: prefer one or two semantic levels, with a third level only for genuinely repeated or complex structures.

## Scope

Use this skill for:

- Figma layer naming and selected-node hierarchy cleanup.
- Semantic group/frame names inside frames, components, review designs, and local explorations.
- Reducing generic names like `Frame`, `Group`, `Rectangle`, `Vector`, `Layer`, `Copy`, and autogenerated labels.
- Handoff readability for existing designs without changing the rendered design.
- Minor grouping or flattening when it improves readability and preserves layout behavior.

Do not use this skill for:

- Component-library promotion, global component architecture, broad relinking, or variant contract redesign.
- Product-screen concept design, feature-flow design, copywriting, or visual redesign.
- Design Language tokens, typography systems, color systems, or dark-mode values.
- Auto Layout repair beyond preserving existing behavior; route sizing issues to the Auto Layout specialist.
- Marketing visuals, store screenshots, landing pages, backend, mobile, frontend implementation, or PM planning.

## Naming And Grouping Rules

- Use names that describe UI role, not visual implementation details alone.
- Prefer names such as `Header`, `Content`, `Body`, `Actions`, `Icon Container`, `Text Stack`, `Metadata Row`, `Card Body`, `Trailing Action`, and `Empty State`.
- Do not group a single child unless the group clarifies a semantic role, protects component boundaries, or preserves Auto Layout behavior.
- Keep raw primitives inside icons or illustrations only when renaming every vector would add noise.
- Avoid vague names like `Wrapper`, `Container`, or `Group` unless paired with a clear role, such as `CTA Container`.

## Figma Workflow

- Follow the required Figma execution skill before any Figma write action.
- Inspect source hierarchy, component instance boundaries, Auto Layout behavior, and nearby naming conventions before mutation.
- Rename and regroup the smallest useful scope first, especially when working inside product screens.
- Do not detach component instances just to rename internal layers unless explicitly requested.
- Validate that the rendered design still matches the source after hierarchy cleanup.

## QA Rules

- Before reporting completion, check that important layers no longer have generic names.
- Confirm grouping depth is practical and not over-nested.
- Confirm no visual layout, token binding, component connection, copy, or Auto Layout behavior changed unexpectedly.
- Report the cleaned frames/components and any areas intentionally left unchanged, such as raw icon vectors or locked component instances.
