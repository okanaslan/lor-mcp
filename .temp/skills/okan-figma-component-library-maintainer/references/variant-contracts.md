# Variant Contracts

Use this when cleaning component names, component sets, and variant properties.

## Naming Rules

- Use stable semantic names: `Card / Task`, `Card / Routine`, `Card / Habit`, `Card / Limit`, `Header / Screen`, `Header / Filter`, `Tab Bar`.
- Keep feature-specific names only when reuse is unlikely, such as `Inbox / Task Action Cell`.
- Avoid unclear names such as `Local/Button 3`, `Frame 18`, `Source=Source8`, or duplicated component names with no semantic difference.

## Variant Properties

Prefer readable properties:

- `State=Default|Loading|Empty|Disabled|Selected|Error|Success`
- `Priority=Critical|High|Medium|Low`
- `Type=Morning|Evening|Fallback`
- `Active=Inbox|Tasks|Routines|Habits|Limits`
- `Size=Small|Default|Large`

## Cleanup Rules

- Preserve existing visual variants when renaming; do not collapse distinct states unless requested.
- Convert autogenerated properties to semantic properties only when the state contract is obvious.
- Do not combine unrelated axes if product behavior treats them as separate states.
- Keep loading/skeleton variants neutral unless the component contract intentionally carries semantic color.
- If a variant contract is ambiguous, leave it unchanged and report the ambiguity instead of inventing behavior.
