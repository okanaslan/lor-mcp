# Figma Component Library Maintainer

Use Figma as the source of truth for reusable app UI components.

## Operating Model

- Start by inspecting the existing `Components` page, component sets, local feature-page masters, and representative instances before mutating anything.
- Preserve visuals, product copy, spacing, and flow behavior unless the user explicitly asks for redesign.
- Promote shared patterns into `Components` when they are reused or clearly intended to be reused.
- Prefer moving existing masters so current instances remain connected; replace duplicates only when the intended global component is unambiguous.
- Normalize names and variants to make component usage readable for designers and implementers.

## Scope

Use this skill for:

- Shared component masters and component sets.
- Local-to-global component promotion.
- Component naming, layer naming, variant names, and component property cleanup.
- Obvious instance relinking and duplicate component cleanup.
- Component QA for layout, variants, missing fonts, clipped text, detached instances, and visual stability.

Do not use this skill for:

- Design Language foundations, tokens, color systems, typography systems, or dark-mode variable values.
- New product screen design, feature flow design, or onboarding-specific flow work.
- Marketing visuals, store assets, landing pages, or public-site design.
- Backend, mobile, frontend implementation, billing logic, analytics, or PM planning.

## Default Maintenance Rules

- Promote + relink by default; do not perform aggressive full-screen refactors unless explicitly requested.
- Keep feature-local components local when reuse is unlikely.
- Rename unclear `Local/*`, autogenerated, or generic component names into stable semantic names.
- Use readable variant properties such as `State`, `Priority`, `Active`, `Type`, or `Size` when the contract is obvious.
- Avoid broken autogenerated variant labels when a clear semantic property exists.
- Do not create new token families or change variables; route that work to the design-system curator skill.

## Figma Workflow

- Follow the required Figma execution skill before any Figma write action.
- Inspect component masters, instances, page placement, and variant properties before changing them.
- Move canonical shared masters to `Components` where possible so existing instances remain connected.
- Replace duplicated local frames only when there is a clear one-to-one component match.
- Keep source frames and product screens visually unchanged unless the user requests a visual update.

## QA Rules

- Before reporting completion, check for clipped text, collapsed frames, missing fonts, unexpected detaches, duplicate masters, broken variant names, and unintended visual changes.
- Verify Auto Layout sizing: fixed for deliberate component dimensions, `Hug contents` for content-driven rows/cards/chips, and `Fill container` only where stretch behavior is intended.
- Return or report the renamed/promoted components, affected pages, and any skipped risky candidates.
