# NativeWind Best Practices Agent

Use this skill to keep NativeWind styling in Expo and React Native apps correct, maintainable, statically detectable, token-aligned, and version-aware.

Use `$okan-tailwindcss-usage-agent` for browser Tailwind work. Use `$okan-figma-to-mobile-implementation-agent` for one-off Figma screen implementation. Use `$okan-mobile-design-system-sync-agent` for broad mobile token or component-system sync. Use `$okan-react-native-performance-audit-agent` for performance-specific audits.

## Operating Model

- Inspect the target app before advising or editing.
- Identify NativeWind and Tailwind versions, Expo/RN version, setup files, CSS entrypoint, content globs, theme tokens, shared primitives, class composition helpers, and third-party components.
- Use current NativeWind docs through Context7 when setup syntax, version behavior, Metro/Babel config, dark mode, CSS variables, `cssInterop`, `remapProps`, or generated types may have changed.
- Prefer existing semantic tokens, app primitives, and explicit component variants over raw colors, duplicated utility strings, or one-off abstractions.
- Keep class names statically detectable by NativeWind/Tailwind. Do not build partial class names dynamically.
- Recommend or implement focused fixes only; avoid broad visual rewrites or design-system migrations unless explicitly requested.

## What To Inspect

Read `references/nativewind-setup-audit.md` when auditing setup or diagnosing NativeWind not applying styles.

Inspect at minimum:

- `package.json` for `nativewind`, `tailwindcss`, Expo, React Native, and styling helper versions.
- `tailwind.config.*` for `nativewind/preset`, content paths, theme extension, plugins, and token mappings.
- `global.css` or the configured CSS input file for Tailwind layers/imports and variable declarations.
- `babel.config.*` for Expo JSX import source and `nativewind/babel` where required by the installed version.
- `metro.config.*` for `withNativeWind` and CSS input path.
- NativeWind TypeScript env file and generated type behavior.
- Theme provider, color-scheme flow, and any runtime CSS variable usage.
- Shared primitives and third-party components that need `className`, `cssInterop`, or `remapProps` support.

## Class Usage Rules

Read `references/class-composition-rules.md` for component review, variant cleanup, or dynamic-class bugs.

Core rules:

- Keep complete utility classes in source, such as `bg-success` or `text-danger`, not template-built fragments like `bg-${tone}`.
- Use explicit maps for tone, size, status, state, density, and platform variants.
- Keep conditional classes tied to clear UI state: selected, disabled, active, pressed, loading, invalid, expanded, open, or status.
- Prefer semantic token utilities such as `bg-panel-raised`, `border-line`, and `text-muted` when the app defines them.
- Use arbitrary values only for real design constraints that do not deserve reusable tokens yet.
- Avoid mixing large inline `style` objects with NativeWind classes unless the value is runtime-only or unsupported by utilities.
- Preserve accessibility and state props when refactoring styled controls.

## Theme, Interop, And Dark Mode

Read `references/theme-interop-dark-mode.md` when working with colors, variables, dark mode, third-party components, or NativeWind interop APIs.

Default rules:

- Map reusable colors, spacing, radii, and typography through app tokens before adding raw utilities.
- Keep light and dark styles symmetrical when a surface/text/border color changes.
- Use NativeWind CSS variable APIs such as `vars` only when runtime theming needs them; otherwise prefer static semantic tokens.
- Use `cssInterop` or `remapProps` for third-party components that do not accept `className` on the actual native style prop.
- Do not assume web Tailwind behavior automatically works on native; verify platform support and fallback behavior.

## Troubleshooting Flow

1. Verify installed versions and whether the app uses NativeWind v4, v5, or another major setup.
2. Verify Metro `withNativeWind` input and the CSS file location/import path.
3. Verify Babel setup and whether JSX import source or `nativewind/babel` is required for that version.
4. Verify `tailwind.config.*` content paths include every source folder where classes are written.
5. Verify complete classes exist in source and are not generated by string interpolation.
6. Verify the target component actually receives style via `className`, `style`, or an interop mapping.
7. Clear caches or rebuild only after config/setup changes make that necessary.

## Verification Expectations

- For code changes, run the repo's relevant typecheck, lint, tests, and app-specific validation commands.
- For setup changes, restart Metro with cache reset or rebuild a dev client when native/config changes require it.
- For theme/dark-mode changes, manually check light and dark mode on at least one iOS or Android target when feasible.
- For third-party component interop, verify the styled prop reaches the underlying native element.
- Report skipped checks, assumptions, version-specific doc source, and remaining device-only risks.

- Manual device checks still needed.
- Any version-specific caveats from current docs.
