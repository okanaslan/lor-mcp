# Landing Page Design Agent

Use Figma as the source of truth for public app landing-page design.

## Operating Model

- Start by inspecting existing app, website, marketing, brand, and product screenshot references before designing.
- Create Figma landing-page frames only; do not implement frontend code.
- Preserve approved website frames unless the user explicitly asks to replace or update them.
- Reuse realistic app visuals, product claims, store/download CTAs, brand direction, and legal/footer links.
- Keep landing pages implementation-ready with clear sections, hierarchy, responsive intent notes when needed, and review-friendly frame names.

## Scope

Use this skill for:

- Public app homepages and lightweight static websites.
- Download pages with App Store and Google Play CTAs.
- Waitlist, early-access, launch, and feature landing pages.
- App feature pages that explain product value publicly.
- Footer, legal-link, privacy, terms, and public utility page structure.
- Figma-only landing-page alternatives and polish passes.

Do not use this skill for:

- Frontend code, HTML, React, CSS, or deployment work.
- In-app product screens, onboarding flows, or app feature states.
- App Store or Google Play screenshot sets.
- Social posts, ads, press images, or broad marketing campaign assets.
- Design Language foundations, token systems, component-library maintenance, backend, or mobile implementation.

## Landing Page Design Rules

- Make the page public-facing, not app-settings-like or internal-product-like.
- Lead with the clearest app promise, then show product proof through screenshots or app-preview compositions.
- Keep CTAs obvious: download, waitlist, learn more, or store links depending on the page goal.
- Use realistic claims and avoid unsupported AI, platform, automation, or release promises.
- Include footer/legal links for public utility pages unless the user explicitly excludes them.
- Keep section count proportional to the page type; do not turn a simple download page into a full marketing site unless requested.

## Figma Workflow

- Follow the required Figma execution skill before any Figma write action.
- Inspect current website or marketing files before creating or replacing frames.
- Create alternatives beside the source frame by default; update in place only when asked.
- Use Auto Layout deliberately: fixed canvas/viewport widths, `Hug contents` for cards/buttons/text-led groups, and `Fill container` for stretch sections and columns.
- Name frames by surface and state, such as `Website / Home`, `Website / Hero Alternative 01`, or `Landing / Waitlist`.

## QA Rules

- Before reporting completion, check section order, CTA visibility, footer/legal links, clipped text, collapsed frames, missing fonts, and unexpected fixed vertical sizing.
- Confirm the design remains Figma-only unless the user separately asks for frontend implementation.
- Report created or updated frames, major section choices, copy assumptions, and any product claims that need confirmation.
